How this Password Generator works

Determinism

The generator is fully deterministic: the same inputs (master password, domain, user, diversificator, and character-set settings) always produce the same output password. There is no random state and nothing is stored anywhere — the password is recomputed each time from your inputs.

This means you do not need a password vault, sync service, or backup. As long as you remember your master password and your settings, you can regenerate any of your passwords on any device that runs this page.

Security guarantees

Only the master password is secret. Everything else is safe to store, share, or sync in plaintext:

  • The pg.html file itself contains no secrets and can be published, copied, or sent to anyone — anyone running it produces their own passwords from their own master.
  • The domain, user, and diversificator values can be written down or kept in any list-keeping app without protection. They function as per-site labels, not as secrets.
  • The character-set settings (which characters and how many of each) can also be saved or shared freely. They affect the shape of the output but not its secrecy.
  • The generated password for one site does not help an attacker derive the password for any other site — each site's password depends on the master combined with that site's unique label, and each section of each password is produced under a freshly-derived permutation.

An attacker who knows your domain/user/diversificator/settings — and even one or more passwords you generated — still has to guess your master password by brute force. The pipeline is intentionally heavy (each password generation runs hundreds of thousands of small block operations), so each guess costs the attacker the same work as it costs you. Pick a long master password.

The algorithm in detail

Given inputs master, domain, user, and optional div:

Step 1. Compute A = MD5(master). This 128-bit value is used as the AES-128 key in the next step.
Step 2. Build B = MD5(domain) ‖ MD5(user), and append MD5(div) if div is non-empty. B is therefore 32 or 48 bytes (two or three 128-bit blocks).
Step 3. Encrypt B with AES-128 in CBC mode under key A, using a zero IV. Each block is XOR'd with the previous ciphertext block before encryption.
Step 4. Take the last ciphertext block as C — the per-site 128-bit derived key. Because of CBC chaining, C depends on every byte of every input.
Step 5. Initialize a TinyPRP cipher (a small AES-style permutation on 16-bit values) with key C. TinyPRP plus cycle walking gives us a keyed pseudo-random permutation on any range [0, n).
Step 6. For each character-class section (capitals, lowercase, digits, specials), if its count is > 0:
  • Shuffle its character set into reSeed using TinyPRP.
  • Append the first count characters of reSeed to the working password D.
  • Update C := C ⊕ MD5(reSeed) and re-key TinyPRP with the new C. This guarantees that the next section (and the final shuffle below) uses a different permutation. Knowing the permutation used for one section reveals nothing about the others.
Step 7. Shuffle the assembled string D one final time with the latest TinyPRP key. The result is the final password.

Why MD5's weaknesses don't matter here

The design uses MD5 as a fast deterministic compression function, not as a security primitive. Replacing it with SHA-256 would make no practical difference to the attacker's task.

In more detail: MD5 is famously broken for collision resistance — it is feasible to construct two different messages with the same MD5 hash. That class of attack is not relevant to this generator, because:

  • We never compare two different MD5 values for equality, nor do we accept arbitrary inputs from an attacker. Collision attacks need the attacker to choose both messages.
  • MD5 is used here purely as a deterministic hash — to compress variable-length inputs (master, domain, user, diversificator, intermediate reSeed values) into fixed-length 128-bit values that feed into AES and TinyPRP.
  • To invert the pipeline an attacker must recover the master from observed outputs — a preimage problem against the AES + TinyPRP stack, not a collision problem against MD5.
  • MD5's preimage and second-preimage resistance, while reduced, remains computationally infeasible (no practical attack is known). Combined with the brute-force cost of each pipeline run, the construction is secure for its purpose.
